# Resume Tailor

Produce targeted, honest resume variants instead of generic polishing.

## Core workflow

1. Read the source resume.
2. Read the target JD.
3. Read the fit analysis if available.
4. Decide what to emphasize, compress, or reorder.
5. Rewrite only within the bounds of truth.
6. Save the result as a distinct version tied to the target role.

## Priorities for this user

Emphasize:

- engineering ownership instead of pure competition language
- autonomous systems understanding across perception / localization / planning / control interfaces
- C++, Python, Linux, ROS, OpenCV, CAN, Git, system integration
- leadership and coordination in student racing / driverless team work
- DLIO optimization as evidence of initiative and project depth

Reduce or compress:

- weakly relevant details that dilute the target role
- generic soft-skill wording without evidence
- repeated statements that do not improve match quality

## Hard rules

- Do not fabricate internships, companies, metrics, or shipped products.
- Keep claims auditable from the user's real history.
- Prefer sharper wording over exaggerated wording.
- If a JD asks for something the user lacks, do not fake it; position adjacent evidence instead.

## Versioning guidance

Each tailored resume should include or be associated with:

- company
- role title
- city
- source JD link or identifier
- version date
- change summary

## Output sections

Return:

1. tailored summary / headline
2. revised project bullets
3. revised skill ordering
4. notable changes made
5. unresolved gaps that resume wording cannot fix

## Tone

Write like a strong early-career engineering candidate, not like a flashy marketer.
